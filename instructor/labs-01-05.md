# Instructor Guide: Labs 1–5

## Delivery model

Run the sequence as five progressive sessions. Begin each session with a working behavior demonstration, let learners predict the system boundary, then expose the starter tests. Preserve at least twenty minutes for the break/fix challenge and debrief.

| Lab | Suggested time | Demonstration | Required checkpoint |
| --- | ---: | --- | --- |
| 1 | 90 min | Recorded and schema-valid incident assessment | Learner explains why valid JSON can still be wrong |
| 2 | 90 min | Bounded search and non-executing proposal | Fixture remains unchanged after proposal creation |
| 3 | 120 min | Health, initialize, list tools, call tool | Learner identifies the failing protocol layer |
| 4 | 120 min | Direct client before model client | No-tool and forbidden-tool cases are visible in traces |
| 5 | 150 min | 401, under-scoped failure, allowed proposal | Learner separates identity, scope, and approval concepts |

## Facilitation checkpoints

### Lab 1

- Ask which constraints belong in prompts and which belong in code.
- Compare refusal, incomplete response, transport error, and schema-validation failure.
- Do not require live API access; recorded mode is the baseline.

### Lab 2

- Challenge any arbitrary URL, method, header, or body parameter.
- Ask learners to prove that the proposal did not mutate the incident.
- Discuss why canonicalization must happen before hashing.

### Lab 3

- Test health before protocol initialization.
- Inspect generated input and output schemas rather than trusting function signatures.
- Seed one wrong endpoint or oversized limit and require evidence before a fix.

### Lab 4

- Keep direct protocol tests and model-selection tests separate.
- Require one prompt that uses no tool and one that requests an unavailable capability.
- Explain why hosted OpenAI services cannot call an undeployed localhost MCP server.

### Lab 5

- Never display real credentials in shared screens or recordings.
- Compare missing token, invalid token, valid-but-under-scoped token, and allowed token.
- Reiterate that the local issuer is a deterministic test fixture, not production identity infrastructure.

## Common misconceptions

- Structured output guarantees schema conformance, not factual correctness.
- MCP does not replace application authorization, validation, or retries.
- A discovered tool is not automatically an authorized tool.
- Authorization does not imply consent for a specific side effect.
- A successful health check does not prove tool discovery or backing-service health.
- A model should not be used to debug a protocol that has not passed direct tests.
